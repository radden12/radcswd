# GGSELBot

Самодостаточный Telegram-бот, который автоматически создаёт заказы в
[SMMPrime](https://smmprime.com/) при покупках на
[GGSEL](https://ggsel.com/) (Digiseller).

Это **полностью отдельный** бот — никакого FunPayCardinal не нужно.
Поведение, тексты, кнопки и состояния максимально повторяют исходный
плагин `smmprime_plugin_v1_0_2.py`.

```
GGSELBot/
├── bot.py            ← одна точка входа
├── config.json       ← создаётся при первом запуске
├── bindings.json     ← создаётся при добавлении первой связки
├── storage/
│   ├── pending_orders.json
│   ├── orders.log
│   └── bot.log
├── requirements.txt
├── start.bat / start.sh
└── README.md
```

## Запуск

### Windows

1. Поставить Python 3.11+ (`Add Python to PATH`).
2. Дважды кликнуть `start.bat`.
3. Откроется командная строка — бот спросит **только Telegram Bot Token**.
4. Введите токен от `@BotFather`. Всё остальное — внутри бота.

### Linux / macOS

```bash
chmod +x start.sh
./start.sh
```

### Ручной запуск

```bash
python3 -m venv venv && source venv/bin/activate
pip install -r requirements.txt
python bot.py
```

## Настройка через Telegram

1. Откройте бота → `/start`. Ваш user_id автоматически становится
   первым админом.
2. **🔑 API-ключ SMMPrime** → пришлите ключ (из ЛК SMMPrime → Settings → API).
3. **🔑 GGSEL seller+api** → пришлите `seller_id`, потом `api_key`
   (ЛК GGSEL → API / «Для разработчиков»).
4. **💰 Баланс** — должен вернуть число.
5. **📃 Список услуг** — выпишите `service_id` нужной услуги SMMPrime.
6. **🛒 Связки → ➕ Добавить связку**:
   - `GGSEL lot_id` (числовой ID товара GGSEL);
   - название (для удобства);
   - `service_id` из шага 5.
7. По умолчанию у новой связки **DRY-RUN ВКЛ** (реальный заказ не идёт).
8. Сделайте тестовую покупку — проверьте, что бот:
   - попросил ссылку;
   - после ссылки — показал сводку «Ссылка / Количество / Услуга»;
   - на «Да» в DRY-RUN отправил шаблон dry_run, реального POST не было.
9. В карточке связки переключите **Dry-run: 🟡 → ⚪** — боевой режим.

## State-machine (как в оригинале)

```
GGSEL: новый заказ
        │
        ▼
[waiting_for_link] ── ссылка ─▶ [waiting_for_confirm] ── «Да» ─▶  create_smm_order
        ▲                            │
        │                       «Отмена»
        └────────────────────────────┘
                                     │
                                     ▼ dry_run ?
                       ┌──── да ─────┴───── нет ─────┐
                       ▼                              ▼
                [dry_run_done]                  SMMPrime add_order
                                                      │
                                            ┌─────────┴─────────┐
                                            ▼                   ▼
                                     [smm_created]          [failed]
```

### Защита от двойных заказов

`_PROCESSING: set[str]` + `_PENDING_LOCK: RLock` гарантируют, что
`smm.add_order` не вызовется дважды для одного `ggsel_order_id`,
даже если события прилетели одновременно.

### Идемпотентность

Если покупатель повторно пишет «Да» по уже оформленному заказу —
бот ответит шаблоном `already_done` и второй POST не пойдёт.

## Шаблоны сообщений (полностью повторяют оригинал)

| Ключ | Когда отправляется | Переменные |
|------|--------------------|-----------|
| `ask_link` | После нового заказа | `{buyer_username}`, `{quantity}`, `{lot_id}` |
| `confirm` | После получения ссылки | `{link}`, `{quantity}`, `{service_id}` |
| `cancelled` | Покупатель написал «Отмена» | — |
| `not_link` | Прислали не-ссылку | — |
| `not_confirm` | Не «Да» и не «Отмена» | — |
| `already_done` | Повторная попытка | — |
| `qty_too_small` | quantity < min услуги SMMPrime | `{min}`, `{quantity}`, … |
| `qty_too_large` | quantity > max услуги SMMPrime | `{max}`, `{quantity}`, … |
| `success` | После успешного `add_order` | `{smm_order_id}`, `{link}`, … |
| `error` | После ошибки SMMPrime | — |
| `dry_run` | В тестовом режиме на «Да» | — |

Все шаблоны редактируются:
- глобально — кнопки 💬 в главном меню;
- per-связка — кнопки ✏ в карточке связки.

Если оба пустые — берётся встроенный дефолт.

## FAQ

**Q: Чем это отличается от GGSELCardinal в соседней папке?**
GGSELCardinal — полный аналог FunPayCardinal-движка с системой плагинов.
GGSELBot — один файл, один процесс, без плагинов. Под одну конкретную
задачу: GGSEL → SMMPrime. Логика и тексты те же самые.

**Q: GGSEL вернул 401 при логине.**
Проверьте `seller_id` (число) и `api_key`. Откройте кнопку «🔑 GGSEL
seller+api» в боте и пришлите ключи заново.

**Q: SMMPrime отклоняет ключ.**
Сравните последние 4 символа в боте (маскировка `abcd****wxyz`) с
ключом в SMMPrime → Settings → API. При расхождении нажмите там
Reset/Generate и обновите ключ в боте.

**Q: Как добавить второго админа?**
В меню → **👤 Админы** → пришлите user_id через запятую.

**Q: Что хранится в pending?**
Любой заказ от GGSEL, попавший под связку. Статусы:
`waiting_for_link`, `waiting_for_confirm`, `dry_run_done`,
`smm_created`, `failed`. Очистить можно из меню «📋 Заказы».
